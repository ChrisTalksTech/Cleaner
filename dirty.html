Before running the EM Cloud deployment scripts, some preparation is required.

AWS Account

An AWS account is required to deploy EM Cloud. An access key must be obtained for an IAM user with sufficient permissions for managing EM Cloud AWS resources.

Accessing EM Cloud Artefacts

EM Cloud environments are built from AWS AMI’s and docker containers. To operate EM Cloud environments access is required to the AMI’s and dockers produced by Product Engineering.

The AWS account used to deploy an EM Cloud environment requires access to the following AMI’s:

  • Rancher Server AMI (used by Rancher control plane)
  • Rancher AMI (used by Rancher worker nodes)
  • Solr Cloud AMI
  • Zookeeper Cloud AMI

The Docker Registry used to host the EM Cloud docker images is Docker Hub (hub.docker.com). To access the images used in EM Cloud a Docker Hub account must be created and then added to the verintdockerhub organisation. See the following link for details: https://kanasoftware.jira.com/wiki/spaces/KMSaaS/pages/852986771/Get+Access+to+the+verintdockerhub+Organization.

The EM Cloud Environment Management project provides a set of scripts that demonstrate how the EM Cloud deployer dockers are invoked. They provide an approach to managing the configuration for multiple EM Cloud environments. These scripts can be used directly or referenced as working examples. The project can be cloned from git@ukgit:EM_cloud-environment-management.

The remainder of the documentation assumes that the environment management scripts are being used.

AWS Resources

Some AWS resources must be created manually before create an EM Cloud environment. Before creating the resources below choose the most appropriate AWS region for hosting the environment. When creating region specific resources, ensure they are created in the chosen region:

  1. Create an S3 Bucket with Versioning feature turned on to save terraform tfstate file
  2. Create a VPC with at least 2 private subnets and 2 public subnets
  3. Create 2 hosted zones for public and private access on Route53 with valid domain that will be accessed by user
  4. Create 2 SSL certificate for each private and public domain using AWS Certificate Manager
  5. Create a keypair to enable access to EC2 instance
